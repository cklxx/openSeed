## Vision

### 10x Check
The 10x version is a research agent that has read every paper in your library, knows the
citation graph cold, tracks how your research interests evolve over time, and proactively
tells you "Paper X from last week contradicts the assumption in Paper Y that you cited in
your draft." It's a collaborator, not a tool.

### Current → Phase 2 → 12-Month Ideal

## Accepted Scope

### Core Agent
1. **Conversational research agent** — Rewrite `assistant.py` for multi-turn, context-aware
   Q&A. Grounded in your library. Citation-grounded answers (e.g., "[Smith 2024, §3.2]").
2. **Library-aware context engine** (`agent/context.py`) — FTS5 + graph walk to find relevant
   papers for any question. Token budget management to fit within model context limits.
   Decomposed into: `_retrieve_papers()`, `_retrieve_memory()`, `_rank_and_truncate()`.
3. **Cross-session research memory** (`agent/memory.py`) — SQLite table `research_memory` in
   library.db. Persists themes, key insights, open questions. Top-K retrieval by relevance.
4. **Research strategy & gap detection** (`agent/strategy.py`) — Analyze graph clusters to
   detect research gaps, suggest reading order, identify under-explored areas.

### UX
5. **Streaming CLI responses** — Rich Live token-by-token display. Buffer tokens for partial
   response recovery on connection drop.
6. **CLI `chat` command** — Multi-turn conversational interface. `--debug` flag shows retrieved
   papers, memory entries, context token count.
7. **Updated `ask` command** — Now uses context engine for grounded answers.

### Platform
8. **MCP Server** (`mcp/server.py`) — stdio transport. Tools: search, graph, summarize, ask,
   memory. Exposes OpenSeed to Claude Desktop / Claude Code.

## Key Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Memory storage | SQLite table in library.db | Consistent with existing storage pattern, FTS5 indexable |
| Memory retrieval | Top-K by relevance | Simple, avoids unbounded growth in context |
| MCP transport | stdio | Standard for local MCP servers, zero config |
| Context overflow | Truncate + retry | BadRequestError must be rescued, not crash |
| Prompt injection | XML boundary tags | Paper content wrapped in `<paper_content>`, marked untrusted |
| Sync/async bridge | Reuse reader.py `_ask()` | DRY — don't invent third approach |
| Streaming recovery | Buffer tokens | Preserve partial response on connection drop |

## Security

- Paper text in context wrapped in XML tags, system prompt treats as untrusted data
- MCP tools are read-heavy; only `ask` triggers Claude API calls
- FTS5 queries use parameterized SQL (existing pattern)
- No new auth surfaces (MCP stdio inherits process permissions)

## Observability

- `--debug` flag on chat/ask shows: retrieved papers, memory entries, token count
- Logging: context query, papers found, context size, turn count, token usage
- Token usage tracked via existing monitor.py integration
- MCP logs tool calls + latency to stderr

## Deferred (Phase 2.5+)

- **Auto-literature review** — from graph clusters, LaTeX output
- **Smart watch suggestions** — from research themes
- **Local embeddings** — semantic search when FTS5 hits limits
